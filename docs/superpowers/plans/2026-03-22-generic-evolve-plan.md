# Generic Evolve Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the agentfolio scaffold work for any single-project repo by replacing hardcoded agentfolio references with config-driven behavior, gated by an interactive onboarding step.

**Architecture:** A config file (`state/evolve_config.md`) generated during interactive CLI onboarding drives all workflow behavior. Every workflow checks for this file immediately after checkout — no file means no execution. The config is injected into each workflow's prompt so Claude adapts behavior (LLM-level adaptation, not bash-level branching).

**Tech Stack:** GitHub Actions YAML, bash, Claude Code CLI (`claude -p`), Astro (profile page)

**Spec:** `docs/superpowers/specs/2026-03-22-generic-evolve-design.md`

**Important notes for implementer:**
- Line numbers are approximate — always locate the step by name, not line number.
- "After the Checkout step" means after the entire `actions/checkout@v4` block including any `with:` parameters.
- deploy.yml has two jobs (`build` and `deploy`). Gate goes in the `build` job only — if build exits, deploy won't run (depends on build via `needs:`).
- All prompt changes are LLM-level: inject the config file and add instructions. Do NOT rewrite bash-level prompt builders.

## Phase 1: Gate Mechanism (all 11 workflows)

### Task 1: Add gate check to all workflows

Add a step immediately after the Checkout step in every workflow that exits if `state/evolve_config.md` doesn't exist.

**Files:**
- Modify: All 11 files in `.github/workflows/`

- [ ] **Step 1: Add gate step to all 11 workflows**

In each workflow, find the `- uses: actions/checkout@v4` step (including any `with:` parameters). Insert this step immediately after it, before any other step:

```yaml
      - name: Check onboarding complete
        run: |
          if [ ! -f state/evolve_config.md ]; then
            echo "::warning::Onboarding not complete. Run: claude 'Set up agentfolio. Follow https://github.com/verkyyi/agentfolio/blob/main/docs/onboarding.md'"
            exit 0
          fi
```

Note for `deploy.yml`: This workflow has two jobs (`build` and `deploy`). Add the gate only to the `build` job. The `deploy` job has `needs: build`, so if build exits early, deploy won't run.

### Task 5: Add periodic re-check and version check to evolve

The spec requires evolve to periodically verify the config still matches the repo, and check for scaffold updates.

**Files:**
- Modify: `.github/workflows/evolve.yml` (prompt section)

- [ ] **Step 1: Add re-check instructions to evolve prompt**

Add a new step to the evolve prompt, after Step 2f (Human Intent Analysis):

```
          ## STEP 2g: Config Re-check (every 7 days)

          Read the "Last recheck" date from the Evolve Config header.
          If more than 7 days have passed since the last recheck:

          1. Scan the repo for structural changes:
             - Check for new dependency files (package.json, requirements.txt, go.mod, Cargo.toml)
               that weren't present during onboarding
             - Check if existing dependency files have new major frameworks
             - Check for new Dockerfile or deploy configs (vercel.json, netlify.toml, etc.)
             - Check for new test frameworks (jest.config, pytest.ini, etc.)
          2. If any significant change detected:
             - Create an issue: "[config] Repo structure may have changed — run CLI session to update evolve config"
             - Label: "needs-review"
          3. If no changes: update "Last recheck" date in state/evolve_config.md
          4. If changes found, do NOT update the date — let the issue prompt re-onboarding

          Only flag changes that affect workflow fit or build commands. Ignore minor additions.
```

- [ ] **Step 2: Add scaffold version check to evolve prompt**

Add after Step 2g:

```
          ## STEP 2h: Scaffold Version Check

          Read the "Version" field from the Evolve Config header.
          Fetch the latest agentfolio release:

          ```bash
          LATEST=$(curl -s -H "Authorization: token $GITHUB_TOKEN" \
            "https://api.github.com/repos/verkyyi/agentfolio/releases/latest" \
            | grep -o '"tag_name":"[^"]*"' | cut -d'"' -f4)
          ```

          If LATEST is newer than the Version field, create an issue:
          "[upgrade] Agentfolio scaffold update available: $LATEST"
          Body: "A new version of the agentfolio scaffold is available. Run a Claude Code
          session to review and apply the changes."
          Label: "needs-review"

          If no release exists or versions match, skip.
```

### Task 6: Rewrite onboarding.md as interactive CLI guide

The onboarding doc becomes instructions that Claude follows during a CLI session — analyze the repo, present findings, ask questions, write config.

**Files:**
- Modify: `docs/onboarding.md`

- [ ] **Step 1: Rewrite onboarding.md**

The doc should instruct Claude to:

1. **Copy scaffold files** from agentfolio (workflows, state, skills, CLAUDE.md)
2. **Analyze the repo**:
   - Read package.json / requirements.txt / go.mod / Cargo.toml / pyproject.toml
   - List file extensions to detect language mix
   - Check for Dockerfile, vercel.json, netlify.toml, fly.toml
   - Check for test config (jest.config, pytest.ini, setup.cfg, Makefile with test target)
   - Read existing README.md for project description
3. **Present findings** to the user:
   - "I detected: [stack]. Build: [command]. Test: [command]."
   - Show workflow fit table (HIGH/MEDIUM/SKIP for each of 11 workflows)
   - Explain what each SKIP means
4. **Ask questions** one at a time:
   - "How do you deploy?" (if no deploy config detected)
   - "Is this a public or private repo?"
   - "These workflows run autonomously with --permission-mode bypassPermissions. This means the agent can execute any command without asking. OK to proceed?"
   - "I'll watch these repos for updates: [list]. Want to add or remove any?"
5. **Write config** to `state/evolve_config.md` with confirmed answers
6. **Initialize state files** (project_state.md, agent_log.md, research_log.md, learned_rules.md)
7. **Commit and push**

Include the exact config file format as a template in the doc.
