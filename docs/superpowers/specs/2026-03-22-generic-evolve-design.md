# Generic Evolve: Make the Scaffold Work for Any Repo

## Problem

Evolve and other workflows are hardcoded for agentfolio. Research sources watch Claude Code harness repos. Analysis steps evaluate an Astro profile page, track agentfolio growth metrics, and monitor agentfolio adopters. Beyond evolve, other workflows also have agentfolio assumptions: coder and reviewer run `npm ci` / `npm run build`, triage references `project:scaffold` labels, analyze hardcodes `APP_NAME: scaffold`. Adopters who add the scaffold to their own repo get irrelevant research, wasted tokens, wrong build commands, and broken steps.

## Solution

Two-phase system: interactive onboarding generates a config file that gates and customizes all autonomous workflows.

## Phase 1: Interactive Onboarding (CLI Session)

Triggered by the user running Claude Code in their repo during setup. NOT triggered by cron or workflow dispatch — requires a human in the loop for confirmation.

### Steps

1. Copy workflows, state files, skills from agentfolio
2. Analyze repo: detect stack (language, frameworks, dependencies), purpose, maturity
3. Estimate workflow fit for each of the 11 workflows:
   - **HIGH** — works as-is, will provide value
   - **MEDIUM** — works but may need tuning
   - **LOW/SKIP** — not applicable to this repo
4. Present estimates to the user. Ask about gaps directly:
   - "How do you deploy?" (if no deploy config detected)
   - "What's your build command?" (detects from package.json scripts, Makefile, etc.)
   - "What's your test command?" (if no test framework detected)
   - "Is this a public or private project?" (determines growth.yml relevance)
   - Highlight that workflows run with `--permission-mode bypassPermissions` — confirm user is OK with autonomous code execution
5. Auto-discover research sources based on detected stack:
   - Identify key dependencies from package.json / requirements.txt / go.mod
   - Map to GitHub repos for those frameworks/libraries
   - Always include `verkyyi/agentfolio` for scaffold updates
6. Write confirmed config to `state/evolve_config.md`
7. Commit and push

Note: SKIP workflows are not disabled or removed. The gate mechanism (Phase 2) prevents them from spending tokens. This avoids modifying workflow YAML during onboarding.

### Config File Format


## Phase 2: Autonomous Evolution (Cron)

### Gate Mechanism

Every workflow checks for `state/evolve_config.md` immediately after checkout, before installing Claude Code or running any other step. If missing, exit immediately with zero tokens and minimal runner time:

```yaml
# Placed immediately after actions/checkout, before all other steps
- name: Check onboarding complete
  run: |
    if [ ! -f state/evolve_config.md ]; then
      echo "Onboarding not complete. Run: claude 'Set up agentfolio in this repo. Follow https://github.com/verkyyi/agentfolio/blob/main/docs/onboarding.md'"
      exit 0
    fi
```

SKIP workflows hit this gate and exit. They still trigger on cron (consuming ~20s of runner time) but never install Claude Code or spend API tokens.

### Prompt Adaptation Strategy

**LLM-level adaptation**: The full `state/evolve_config.md` is injected into each workflow's prompt via `$(cat state/evolve_config.md)`. Claude reads the config and adapts its behavior — skipping irrelevant steps, using the correct build/test commands, researching the right sources.

This is simpler than bash-level branching (no dynamic prompt generation) and the token cost of injecting the config (~200 tokens) is negligible on opus/max.

Each workflow prompt includes an instruction block:

```
## Evolve Config (read this — adapt your behavior accordingly)
$(cat state/evolve_config.md)

When a workflow is marked SKIP, do not execute steps related to it.
Use the Build section for install/build/test commands — do NOT assume npm.
Use the Research Sources for Step 1 — do NOT use hardcoded sources.
```

### Evolve Behavior (with config)

Evolve reads `state/evolve_config.md` and:

- **Research**: Fetches commits from auto-discovered sources + `verkyyi/agentfolio`. No hardcoded gstack/ECC/etc unless relevant to the repo.
- **Pipeline health** (Step 2a): Runs as-is — generic, checks workflow failures.
- **Design evaluation** (Step 2b): Three-tier — framework sites get full design review, static HTML sites get content/SEO evaluation, repos with no web presence skip entirely.
- **Growth metrics** (Step 2c): Skipped unless config indicates public repo.
- **Adoption tracking** (Step 2d): Skipped unless config indicates template/scaffold repo.
- **SEO** (Step 2e): Skipped unless config indicates public site.
- **Human intent analysis** (Step 2f): Runs as-is — generic.
- **Scaffold update check**: Compares config `Version` field against latest agentfolio release. If newer release exists, creates an issue suggesting an upgrade.

### Periodic Re-check (every 7 days)

Evolve tracks a `last_recheck` timestamp in `state/evolve_config.md`. Every 7 days, it re-scans the repo and compares against the config:

**Triggers for mismatch detection:**
- New dependency file appeared (e.g., `requirements.txt` added to a Node repo → Python added)
- Existing dependency file has new major frameworks (e.g., `package.json` now includes `next`)
- New `Dockerfile` or deploy config appeared
- Test framework appeared where none was configured

**On mismatch**: Creates an issue: "Your repo structure may have changed since onboarding. Run a CLI session to update your evolve config." Does NOT auto-update the config — human confirmation required.

**Avoiding false positives**: Only flags changes that affect workflow fit or build commands. Minor dependency additions (patch versions, dev tools) are ignored.

### Other Workflows (with config)

- **coder**: Reads config for build/test commands. Uses `install` command instead of `npm ci`. Runs `test` command instead of `npm run build`. Falls back to skip if no command configured.
- **reviewer**: Same — reads config build/test commands. Runs the configured test command to validate PRs.
- **triage**: Reads config for project structure. Uses `project:<name>` label from config instead of hardcoded `project:scaffold`.
- **watcher**: Generic pipeline health checks. Reads config to know which workflows are SKIP (doesn't flag their "failures" as problems).
- **analyze**: Reads config for project name instead of hardcoding `scaffold`.
- **claude-task**: Reads config for build commands.
- **deploy**: Gate check handles SKIP. If HIGH, runs as-is.
- **growth**: Gate check handles SKIP. If HIGH, runs as-is.
- **feedback-learner**: Runs if MEDIUM or HIGH.
- **discover**: Not needed for single-project repos. SKIP by default.

### Project Structure

Adopter repos are single-project. The `apps/` directory and `APP_NAME` resolution are not used. CLAUDE.md lives at the repo root. Workflows that currently read `apps/${APP_NAME}/CLAUDE.md` are updated to read `./CLAUDE.md` based on the config.

## Scope

### Changes Required

1. **New file**: `state/evolve_config.md` — generated by onboarding, read by all workflows
2. **All 11 workflows**: Add gate check step immediately after checkout
3. **All 11 workflows**: Inject config into prompts, add instruction block for adaptation
4. **evolve.yml prompt**: Replace hardcoded sources/steps with config-driven instructions
5. **coder.yml, reviewer.yml, claude-task.yml**: Replace `npm ci` / `npm run build` with config-driven commands
6. **triage.yml**: Replace `project:scaffold` with config-driven label
7. **analyze.yml**: Remove hardcoded `APP_NAME: scaffold`, read project name from config
8. **docs/onboarding.md**: Rewrite as interactive CLI session guide with repo analysis, fit estimation, Q&A, and config generation
9. **Profile page**: Update "Add to Your Repo" section to reflect the interactive onboarding

### Not Changed

- Workflow triggers (cron schedules, event triggers) — unchanged
- State file format (agent_log, research_log, etc.) — unchanged
- Skills — unchanged
- CLAUDE.md autonomy rules — unchanged (adopters replace with their own during onboarding)

## Risk

- **Auto-discovery accuracy**: Claude may misjudge the stack or pick irrelevant sources. Mitigated by human confirmation during onboarding.
- **Config staleness**: Repo changes but config doesn't. Mitigated by 7-day re-check + issue creation.
- **Gate failure**: If `state/evolve_config.md` is accidentally deleted, all workflows stop. This is by design — forces re-onboarding rather than running blind. Documented in config file header.
- **Runner minutes for SKIP workflows**: Cron-triggered SKIP workflows still consume ~20s of runner time per trigger. Negligible for public repos (unlimited minutes). For private repos with many SKIP workflows, users can disable specific workflow schedules in GitHub UI.
- **Prompt adaptation reliability**: Claude may not perfectly follow config instructions every time. Opus/max gives the best instruction following. Critical commands (build, test) should have explicit fallback behavior (skip if command not configured, don't assume defaults).
