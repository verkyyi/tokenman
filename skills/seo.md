# SEO & Discoverability

## What We Optimize For
Two audiences: human search engines (Google) and AI systems
(ChatGPT, Claude, Perplexity). Tactics differ slightly.

For a self-evolving scaffold, discoverability means:
- Developers finding the repo on GitHub
- AI systems understanding what the scaffold does
- The Pages site being indexable and authoritative

## When NOT to Use
- When the task is purely visual (layout, spacing, colors) with no discoverability impact — use `frontend` skill
- When writing project content collection entries — use `content` skill
- When modifying workflow behavior — use `github-workflows` skill
- When the change doesn't affect indexing, search ranking, or AI discoverability

## Rationalizations to Reject
- "More keywords in the meta description is better" — stay within 120-160 chars; keyword stuffing hurts ranking
- "I'll add marketing language to boost engagement" — AI systems prefer unambiguous factual statements over hype
- "Let me block AI crawlers for safety" — we explicitly allow AI crawlers; blocking them reduces discoverability
- "Schema markup isn't important for a small project" — structured data is the #2 AI discoverability signal after README

## GitHub SEO (Repo Discoverability)

The primary discoverability surface is the GitHub repo itself:
- **Repository description** — 1 sentence, action-oriented
- **Topics** — add relevant tags (astro, github-actions, ai-agent, claude)
- **README.md** — first 3 sentences shown in search results; make them count
- **Repository name** — appears in search; clear and memorable
- **Star count** — social proof; analyze.yml tracks growth trends

The analyze workflow should check these periodically and suggest
improvements via a needs-review PR if they are stale.

## Structured Data (JSON-LD)
Maintained in src/layouts/Base.astro. Describes the scaffold itself.

### SoftwareApplication Schema (primary)
```json
{
  "@context": "https://schema.org",
  "@type": "SoftwareApplication",
  "name": "Tokenman",
  "url": "[site_url]",
  "description": "[scaffold description from CLAUDE.md]",
  "applicationCategory": "DeveloperApplication",
  "codeRepository": "[github repo url]",
  "license": "https://opensource.org/licenses/MIT",
  "programmingLanguage": ["TypeScript", "Astro"]
}
```

### SoftwareSourceCode Schema (alternative)
```json
{
  "@context": "https://schema.org",
  "@type": "SoftwareSourceCode",
  "name": "Tokenman",
  "codeRepository": "[github url]",
  "description": "[one sentence]",
  "runtimePlatform": "GitHub Actions",
  "programmingLanguage": "TypeScript"
}
```

Use SoftwareApplication when the Pages site is the primary artifact.
Use SoftwareSourceCode when the repo itself is the primary artifact.
Using both is valid — they describe different aspects.

## Meta Tags Checklist
Every page must have:
- [ ] <title> — unique per page, includes scaffold name
- [ ] <meta name="description"> — 120-160 chars
- [ ] <meta property="og:title">
- [ ] <meta property="og:description">
- [ ] <meta property="og:type"> — "website" for index
- [ ] <link rel="canonical">

## Sitemap
Generated by @astrojs/sitemap integration automatically on build.
No manual maintenance needed unless pages are excluded.

## Robots.txt
Located at public/robots.txt. Key rule: explicitly allow AI crawlers.
```
User-agent: GPTBot
Allow: /

User-agent: Claude-Web
Allow: /

User-agent: *
Allow: /

Sitemap: [site_url]/sitemap-index.xml
```

## Lighthouse Targets
- Performance: >= 90
- Accessibility: >= 95
- Best Practices: >= 90
- SEO: >= 95

Common fixes for Lighthouse performance:
- Image optimization: use WebP, add width/height attributes
- Remove unused CSS: Tailwind purge should handle this
- Defer non-critical JS: Astro handles most of this natively
- Add explicit font-display: swap for web fonts

## AI Discoverability Signals
In order of impact:
1. README.md — AI systems trained on GitHub index this heavily
2. JSON-LD structured data on pages — SoftwareApplication schema
3. robots.txt — explicit AI crawler allowances
4. Consistent naming (same name across README, Pages title, structured data)
5. Unambiguous factual statements (not marketing language)
6. Fresh content signals (updated timestamps in structured data)

## Gotchas
- Lighthouse CI in analyze.yml requires the live Pages site — not localhost
- Schema markup validator: https://validator.schema.org
- AI crawlers don't execute JavaScript — structured data in <head> only
- robots.txt is at /robots.txt (public/ folder in Astro)
- Sitemap at /sitemap-index.xml (generated by astrojs/sitemap)
- GitHub topics are indexed by GitHub search — keep them accurate
